🎯Visão Geral do Projeto

O Sistema de Gestão Acadêmica é uma aplicação educacional que demonstra a implementação completa de uma arquitetura de microsserviços com padrões modernos de desenvolvimento distribuído.

Objetivo Principal

Este projeto foi desenvolvido para fins educacionais, ilustrando como diferentes serviços independentes podem trabalhar juntos de forma sincronizada, mantendo seus próprios bancos de dados e comunicando-se via APIs REST.

O que o Sistema Gerencia

  • 👨‍🎓 Alunos → Cadastro, atualização e remoção de estudantes
  • 📚 Cursos → Administração completa de cursos disponíveis
  • 📋 matrículas → Associação entre alunos e cursos com dados enriquecidos

Diferencial Técnico

O principal diferencial é a integração inteligente entre serviços: ao consultar uma matrícula, o sistema automaticamente agrega dados de alunos e cursos de forma sincronizada, sem compartilhar banco de dados entre os serviços.

🤔 Por que Microsserviços? Em uma arquitetura monolítica tradicional, toda a aplicação seria um único bloco. Com microsserviços, cada funcionalidade é um serviço independente, permitindo escalabilidade, manutenção e evolução independentes.

Backend é Java + Spring Boot

Linguagem de programação compilada, type-safe e orientada a objetos. Versão LTS com melhorias de performance.

Framework que simplifica o desenvolvimento de aplicações Java. Fornece configuração automática e embedded server.

Facilita a persistência de dados. Abstrai a complexidade SQL com mapeamento objeto-relacional (ORM) via Hibernate.

Lombok

Biblioteca que reduz código boilerplate (getters, setters, construtores) através de anotações.

RestTemplate

Cliente HTTP do Spring para comunicação síncrona entre microsserviços. Simples e confiável.

Frontend é React + Node.js

React 18

Biblioteca JavaScript para construção de interfaces reativas. Componentes reutilizáveis e estado gerenciado.

Vite 5

Build tool moderna extremamente Rápida. Substitui o Webpack com inicialização instantânea em dev mode.

Axios

Cliente HTTP para JavaScript. Simplifica requisições aos microsserviços com suporte a promises e interceptors.

React Router 6

Biblioteca de roteamento para SPA (Single Page Application). Permite Navegação entre páginas sem recarregar.

Padrão de Arquitetura: Database per Service

Aspecto Descrição Benefício
Banco Isolado Cada microsserviço possui seu próprio banco de dados Evita acoplamento e permite evolução independente
Sem Compartilhamento serviços não acessam banco de dados um do outro Autonomia total; mudanças não afetam outros serviços
comunicação síncrona RestTemplate para chamadas HTTP entre serviços Simples, direto e fácil de debugar

Fluxo de comunicação: Exemplo Prático

Cenário: Consultar uma matrícula com Dados Completos

  1. Cliente (Frontend) faz: GET /matriculas/1
  2. ms-matricula recebe a requisição
  3. ms-matricula busca em seu banco: matricula_id=1
  4. ms-matricula extrai alunoId=5 e cursoId=3
  5. ms-matricula chama ms-aluno: GET http://localhost:8081/alunos/5
  6. ms-matricula chama ms-curso: GET http://localhost:8082/cursos/3
  7. ms-matricula agrega os dados: matrícula + aluno + curso
  8. Retorna ao cliente uma resposta enriquecida com todos os dados
Resultado: O cliente recebe TUDO em uma resposta, sem precisar fazer múltiplas requisições.

Quando executado, Spring Boot:

  • Conecta ao banco MySQL "aluno_db"
  • Cria a tabela "alunos" automaticamente (se não existir)
  • Escuta requisições HTTP

🏛️model/Aluno.java

Entidade que representa um aluno no banco de dados:

@Entity
@Table(name = "alunos")
@Data
@NoArgsConstructor
@AllArgsConstructor
public class Aluno {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    private String nome;        // Nome do aluno
    private String email;       // Email único do aluno
}

anotações explicadas:

  • @Entity é Indica que é uma entidade JPA mapeada ao banco
  • @Table(name = "alunos") é Nome da tabela no MySQL
  • @Data é Lombok: gera getters, setters, toString, equals
  • @Id é Chave primária
  • @GeneratedValue é Auto-incremento

🗄️repository/AlunoRepository.java

Interface para acesso ao banco de dados:

@Repository
public interface AlunoRepository extends JpaRepository<Aluno, Long> {
    // JpaRepository fornece automaticamente:
    // - findAll() - lista todos
    // - findById(id) - busca por ID
    // - save(aluno) - insere ou atualiza
    // - deleteById(id) - deleta por ID
}

Spring Data JPA gera a implementação automaticamente. não precisa escrever SQL!

⚙️service/AlunoService.java

Camada de Lógica de negócio:

@Service
public class AlunoService {
    @Autowired
    private AlunoRepository repository;
    
    public List<Aluno> listarTodos() {
        return repository.findAll();
    }
    
    public Aluno buscarPorId(Long id) {
        return repository.findById(id)
            .orElseThrow(() -> new RuntimeException("Aluno não encontrado"));
    }
    
    public Aluno criar(Aluno aluno) {
        return repository.save(aluno);
    }
    
    public Aluno atualizar(Long id, Aluno aluno) {
        Aluno existente = buscarPorId(id);
        existente.setNome(aluno.getNome());
        existente.setEmail(aluno.getEmail());
        return repository.save(existente);
    }
    
    public void deletar(Long id) {
        repository.deleteById(id);
    }
}

🎮controller/AlunoController.java

Expõe os endpoints REST:

@RestController
@RequestMapping("/alunos")
public class AlunoController {
    @Autowired
    private AlunoService service;
    
    @GetMapping
    public List<Aluno> listar() {
        return service.listarTodos();
    }
    
    @GetMapping("/{id}")
    public Aluno buscar(@PathVariable Long id) {
        return service.buscarPorId(id);
    }
    
    @PostMapping
    public Aluno criar(@RequestBody Aluno aluno) {
        return service.criar(aluno);
    }
    
    @PutMapping("/{id}")
    public Aluno atualizar(@PathVariable Long id, @RequestBody Aluno aluno) {
        return service.atualizar(id, aluno);
    }
    
    @DeleteMapping("/{id}")
    public void deletar(@PathVariable Long id) {
        service.deletar(id);
    }
}

Fluxo de uma requisição GET /alunos

  1. Frontend faz: GET http://localhost:8081/alunos
  2. Spring roteia para AlunoController.listar()
  3. Controller chama service.listarTodos()
  4. Service chama repository.findAll()
  5. Spring Data JPA executa: SELECT * FROM alunos;
  6. Banco retorna lista de alunos
  7. Service retorna para Controller
  8. Controller converte para JSON e retorna ao frontend

🔄service/MatriculaService.java

Lógica de orquestração (a mágica acontece aqui!):

@Service
public class MatriculaService {
    @Autowired
    private MatriculaRepository repository;
    
    @Autowired
    private RestTemplate restTemplate;
    
    // ? Método que agrega dados de outros serviços
    public MatriculaDetalhe buscarComDetalhes(Long id) {
        // 1. Busca matrícula em seu próprio banco
        Matricula matricula = repository.findById(id)
            .orElseThrow(() -> new RuntimeException("matrícula não encontrada"));
        
        // 2. Chama ms-aluno para pegar dados do aluno
        AlunoInfo alunoInfo = restTemplate.getForObject(
            "http://localhost:8081/alunos/{id}",
            AlunoInfo.class,
            matricula.getAlunoId()
        );
        
        // 3. Chama ms-curso para pegar dados do curso
        CursoInfo cursoInfo = restTemplate.getForObject(
            "http://localhost:8082/cursos/{id}",
            CursoInfo.class,
            matricula.getCursoId()
        );
        
        // 4. Monta e retorna a resposta completa
        return new MatriculaDetalhe(
            matricula.getId(),
            matricula.getAlunoId(),
            matricula.getCursoId(),
            alunoInfo,
            cursoInfo
        );
    }
}

Fluxo de Enriquecimento de Dados

GET /matriculas/1

  1. Controller recebe requisição
  2. Chama: service.buscarComDetalhes(1)
  3. Service busca no banco local: matricula_id=1
  4. Service faz: GET http://localhost:8081/alunos/5
  5. Service faz: GET http://localhost:8082/cursos/3
  6. Service agrega todos os dados
  7. Retorna JSON completo ao cliente

Exemplo de Resposta Enriquecida

🎓Conceitos-Chave Demonstrados

padrões de Arquitetura

🧭 microsserviços

Cada serviço é independente, escalável e responsável por um domínio específico. Falha em um não afeta os outros.

🧭 Database per Service

Cada microsserviço possui seu próprio banco de dados. Evita acoplamento e permite evolução independente.

🧭 REST API

comunicação padronizada via HTTP com verbos corretos (GET, POST, PUT, DELETE) e respostas em JSON.

🧭 Enriquecimento de Dados

Agregação de dados de múltiplas fontes para retornar respostas completas sem múltiplas requisições.

🎯 Separação de Responsabilidades

Controller → Service → Repository: cada camada com sua função clara.

💉 Injeção de Dependência

Spring gerencia automaticamente instâncias de classes (@Autowired), desacoplando a criação de objetos.

padrões de código

  • SOLID: Principios de design orientado a objetos (especialmente SRP e DIP)
  • DAO (Data Access Object): Repository pattern para acesso ao banco
  • DTO (Data Transfer Object): MatriculaDetalhe encapsula dados para transferência
  • MVC (Model-View-Controller): Separação entre model, view (Frontend) e controller

Tecnologias Spring Boot

  • @SpringBootApplication: Anotação que marca classe principal
  • @RestController: Marca classe como controlador REST, retorna JSON automaticamente
  • @Service: Marca classe como serviço (Lógica de negócio)
  • @Repository: Marca interface como acesso a dados
  • @Entity: Marca classe como entidade JPA
  • @Autowired: Injeção automática de dependências

Ferramentas e Frameworks

  • JPA/Hibernate: ORM (Object-Relational Mapping) que mapeia classes Java a tabelas SQL
  • Spring Data JPA: Abstrai ainda mais, gerando queries automaticamente
  • Lombok: Reduz boilerplate com anotações (@Data, @NoArgsConstructor)
  • RestTemplate: Cliente HTTP do Spring para chamar outros serviços
  • React Hooks: useState, useEffect para gerenciar estado no frontend
  • Axios: Promise-based HTTP client para JavaScript
